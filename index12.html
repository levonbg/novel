ChaCha20-Poly1305是一种加密算法，分别是对称加密算法和MAC算法，为保证机密性和完整性，这两种算法必须同时存在
chacha20 对称加密算法， 根据key创造一个chacha，设置iv和seek Crypt= 原始输入异或Keystream
    ChaCha系列流密码，作为salsa密码的改良版，具有更强的抵抗密码分析攻击的特性，“20”表示该算法有20轮的加密计算。
    加密过程即是将密钥流与明文逐字节异或得到密文，反之，解密是将密文再与密钥流做一次异或运算得到明文。
    ARMv7表现是AES的4倍,v8 AES更好
poly1305 MAC算法 ，传入input和key，得到output，确认output和xxx是不是一致来认证
    Poly1305消息认证码，可用于检测消息的完整性和验证消息的真实性，与salsa20或ChaCha20流密码结合使用。
    输入为32字节（256bit）的密钥和任意长度的消息比特流，经过一系列计算生成16字节（128bit）的摘要。
ps：支持aes指令集的cpu，优先aes，否则chacha20（arm v7）

概要设计
1.浏览器根据系统设置，使用代理访问http
2.系统使用了pac，根据pac选择是直连还是代理  // 为了防止所有流量都走代理，立刻被嗅探，使用白名单，名单里的代理，其他直连
3.使用http或者socks5代理                    // 为了解决某些软件不支持socks5
4.localhost的代理服务器收到了请求，通过wss透传给vps    // 流量是wss，像正常的http访问，不需要混淆，自带加密，aes256gcm/chacha20
5.收到应答给浏览器

6.local客户端，请求cdn的域名地址，加随机字符串，可以连到远程的wss    // 前端套个cdn，加速，而且隐藏自己真实ip
7.需要设计多个随机字符串，或多个域名，达到混淆和负载均衡的效果
8.通过wss，自带加密，建立了正常的连接，在wss里，用非对称算法给对方aes密钥    //不用ca证书，不安全
9.所有的socks5请求，都aes加密以后，在wss里传输            // 双保险，外层wss用的是ca证书交换了aes密钥，内层用自己的公钥交换aes密钥

10.remote服务端，是一个web服务器，wss的地址是随机的，只有我自己知道随机地址是什么，对应去建立wss
11.wss有鉴权，socks5自带鉴权够用了
12.建立wss以后，得到了socks5连接，转发请求，桥接，返回应答

http1.1 相对 1.0的优势

1.缓存处理，HTTP1.1则引入了更多的缓存控制策略例如Entity tag，If-Unmodified-Since, If-Match, If-None-Match等更多可供选择的缓存头来控制缓存策略。
2.带宽优化及网络连接的使用，请求头引入了range头域，它允许只请求资源的某个部分，即返回码是206（Partial Content），这样就方便了开发者自由的选择以便于充分利用带宽和连接。
3.错误通知的管理，在HTTP1.1中新增了24个错误状态响应码
4.Host头处理，HTTP1.1的请求消息和响应消息都应支持Host头域
5.长连接，HTTP 1.1支持长连接（PersistentConnection）和请求的流水线（Pipelining）处理。HTTP1.1中默认开启Connection： keep-alive

HTTP/1.x的缺陷

1.连接无法复用，频繁3次握手。加入keep-alive可以复用一部分连接，但域名分片等情况下仍然需要建立多个connection
2.Head-Of-Line Blocking，
3.协议开销大：HTTP/1.x中header内容过大（每次请求header基本不怎么变化），增加了传输的成本。

HTTP2

1.Header压缩
2.流（stream）和多路复用（MultiPlexing）
2.1    流量控制（Flow Control）
2.2 流优先级（Stream Priority）
3.Server Push
tip:WebSocket 是一个双向通信协议，它在握手阶段采用 HTTP/1.1 协议（暂时不支持 HTTP/2）。

ASR
■ MRCP Client发送INVITE消息给MRCP Server请求建立会话，携带MRCP Client侧的SDP；
■ MRCP Server回复200表示请求已经成功接受处理，携带MRCP Server侧的SDP；
■ MRCP Client随后发送ACK消息证实200消息已经收到，至此一个SIP会话成功建立；
■ MRCP Client发送RECOGNIZE消息给MRCP Server请求语音识别，按照MRCP协议规定的格式携带相关的语音识别控制参数，并且指定语法文件路径；
■ MRCP Server接收RECOGNIZE请求，编译语法文件，回复200消息给MRCP Client；
■ MRCP Client此时开始根据之前协商好的SDP，开始源源不断的发送RTP语音流给MRCP Server；
■ MRCP Server接收RTP语音流，当检测到用户开始说话时，发送START-OF-INPUT事件；
■ 当MRCP Server根据语法文件定义得到识别结果时，通过RECOGNITION-COMPLETE事件返回识别结果；
■ MRCP Client发送BYE消息给MRCP Server结束会话；
■ MRCP Server发送200消息给MRCP Client确认结束；
■ MRCP Client通过上述流程获得MRCP Server提供的一次完整语音识别能力。

TTS
■ SPEAK:向服务器端提供文本,启动语音合成(c→s)
■ STOP:如果服务器正在语音合成资源,则停止语音合成与语音流(c→s)
■ PAUSE:通知服务器资源暂停语音合成与语音流(c→s)。
■ RESUME:通知暂停的语音合成资源继续进行语音合成与语音流(c→s)
■ CONTROL:更改语音合成资源相关参数,从而影响合成的语音流(c→s)。
■ SPEAK-COMPLETE: SPEAK请求已经成功处理(s→c)。
■ SPEECH-MARKER:服务器正在处理语音标签时,遇到请求消息头字段 Speech Marker中标记的tag(s→c)。
■ BARGE-IN-OCCURRED:客户端检测到barg-in-able事件或DTMF数字时,发送该消息通知服务器(c→s)。
