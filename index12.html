ChaCha20-Poly1305是一种加密算法，分别是对称加密算法和MAC算法，为保证机密性和完整性，这两种算法必须同时存在
chacha20 对称加密算法， 根据key创造一个chacha，设置iv和seek Crypt= 原始输入异或Keystream
    ChaCha系列流密码，作为salsa密码的改良版，具有更强的抵抗密码分析攻击的特性，“20”表示该算法有20轮的加密计算。
    加密过程即是将密钥流与明文逐字节异或得到密文，反之，解密是将密文再与密钥流做一次异或运算得到明文。
    ARMv7表现是AES的4倍,v8 AES更好
poly1305 MAC算法 ，传入input和key，得到output，确认output和xxx是不是一致来认证
    Poly1305消息认证码，可用于检测消息的完整性和验证消息的真实性，与salsa20或ChaCha20流密码结合使用。
    输入为32字节（256bit）的密钥和任意长度的消息比特流，经过一系列计算生成16字节（128bit）的摘要。
ps：支持aes指令集的cpu，优先aes，否则chacha20（arm v7）

概要设计
1.浏览器根据系统设置，使用代理访问http
2.系统使用了pac，根据pac选择是直连还是代理  // 为了防止所有流量都走代理，立刻被嗅探，使用白名单，名单里的代理，其他直连
3.使用http或者socks5代理                    // 为了解决某些软件不支持socks5
4.localhost的代理服务器收到了请求，通过wss透传给vps    // 流量是wss，像正常的http访问，不需要混淆，自带加密，aes256gcm/chacha20
5.收到应答给浏览器

6.local客户端，请求cdn的域名地址，加随机字符串，可以连到远程的wss    // 前端套个cdn，加速，而且隐藏自己真实ip
7.需要设计多个随机字符串，或多个域名，达到混淆和负载均衡的效果
8.通过wss，自带加密，建立了正常的连接，在wss里，用非对称算法给对方aes密钥    //不用ca证书，不安全
9.所有的socks5请求，都aes加密以后，在wss里传输            // 双保险，外层wss用的是ca证书交换了aes密钥，内层用自己的公钥交换aes密钥

10.remote服务端，是一个web服务器，wss的地址是随机的，只有我自己知道随机地址是什么，对应去建立wss
11.wss有鉴权，socks5自带鉴权够用了
12.建立wss以后，得到了socks5连接，转发请求，桥接，返回应答
